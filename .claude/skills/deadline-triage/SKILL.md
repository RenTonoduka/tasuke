# Deadline Triage

Tasukeの全タスクを俯瞰し、デッドラインと優先度を整理する。Claude本体の判断力で「どれを今やるべきか」を提案し、ユーザー承認後にMCPで一括反映する。

## When To Use

- 「デッドラインを整理したい」「タスク棚卸し」「優先度見直し」と言われた時
- 期限切れタスクが溜まってきて再配分したい時
- 週次・月次の運用リズムで定例実行したい時
- メンバーの負荷分散を見直したい時

## Hard Requirements

- `mcp__tasuke__*` ツール群が利用可能であること（未接続なら `claude mcp add task-core --transport http https://tasuke.app/api/mcp --header "Authorization: Bearer tsk_xxx"` を案内）
- ユーザーが対象ワークスペースのメンバーであること

## 実行手順

### Step 1: コンテキスト収集（並列実行）

以下を **必ず並列で** 取得してトークン効率を最大化:

```
- mcp__tasuke__dashboard         (期限切れ/今日/今週/進行中)
- mcp__tasuke__workspace_stats   (全体の進捗ヘルスチェック)
- mcp__tasuke__project_list      (プロジェクト一覧+セクション)
- mcp__tasuke__my_tasks          (自分のタスクのみ → 個人triageの場合)
- mcp__tasuke__member_list       (担当者解決のため)
- mcp__tasuke__calendar_event_list (任意。次2週間の空き時間を見る場合)
```

### Step 2: 分析（あなたが判断する）

収集データから以下を分析:

1. **期限切れタスク**
   - 今すぐ対応 / 期限を引き伸ばす / 優先度下げて放置 / 削除
   - ステータス IN_PROGRESS で完了予定が立たないものを発見

2. **負荷集中の検出**
   - 同一担当者にP0/P1が今週に集中していないか
   - メンバー間の偏りがあるか

3. **依存関係の整合性**
   - 「Aが終わったらB」の前後関係（タスク説明文・タイトルから推論）
   - 親タスクが進んでいないのに子タスクの期限が近いケース

4. **クリティカルパス**
   - 全体の最終期限（プロジェクトの大型タスクの期限）から逆算
   - 遅延すると後段がドミノ倒しになるタスクをマーク

5. **現実的な再配分**
   - estimatedHours と稼働日数のバランス（カレンダー空き時間が取れるか）
   - 重要度が低いタスクは P3 → 翌週以降に移動

### Step 3: 提案フォーマット

以下のテーブル形式で **必ず3セクション** に分けて提示:

```
## 🚨 緊急対応推奨（今日〜3日）
| タスク | 担当 | 現状期限 → 提案 | 優先度 → 提案 | 理由 |
|---|---|---|---|---|

## ⚠️ 再配分推奨（今週中）
（同上）

## 📅 来週以降に移動
（同上）
```

加えて以下のサマリーを必ず付ける:

```
## 全体ヘルス
- 期限切れ: N件 → 提案後 N件
- 今週P0/P1集中: メンバーAに M件 → メンバーBに振り分け推奨 X件
- クリティカルパス: タスクY → タスクZ（Yが遅れると全体N日遅延）
- 想定リスク: ◯◯
```

### Step 4: 承認フロー

ユーザーに以下を質問:
- 「全部承認」「個別に確認したい」「却下」「もう一度別案」のどれにしますか？

**個別確認の場合:**
- 1件ずつ表示し「承認/スキップ/別の案」を選んでもらう

**全部承認の場合:**
- 「これから次の変更を一括反映します。よろしいですか？」と最終確認

### Step 5: 一括反映

承認された変更を以下のMCPツールで実行:

```
- mcp__tasuke__task_update       (個別更新: dueDate, priority, status)
- mcp__tasuke__task_bulk_update  (5件以上は一括 - actionは status|priority|delete のみ)
- mcp__tasuke__task_assignee_set (担当者変更)
- mcp__tasuke__task_move         (セクション移動: 例 Todo→進行中)
```

実行後、必ず `mcp__tasuke__dashboard` を再取得して **before/after** を表示。

## 判断基準（重要）

### 期限を伸ばすか縮めるか
- **伸ばす:** 着手0% かつ依存先がブロックされていない時
- **そのまま:** 進行中で目処が立っている時
- **縮める:** プロジェクト全体の期限から逆算して必要な時

### 優先度の調整原則
- **P0:** 今日〜明日に絶対に終わらせる必要があるもの（顧客影響・契約期限等）
- **P1:** 今週中に必須
- **P2:** 今月中
- **P3:** いつでも（バックログ）

新規P0は基本作らない。既存P0が増えるなら何かをP1以下に降格する。

### 負荷分散の閾値
- 1メンバーが同一週内にP0を2件以上 → 警告
- 1メンバーがP0+P1合計5件以上 → 警告
- 削減できないなら期限を翌週以降にずらす提案

## ❌ 絶対にやらないこと

- ユーザー承認なしの一括変更
- 削除（DELETE系）の自動実行 — 必ず明示確認
- 完了済み（DONE）タスクへの変更
- 別ワークスペースのタスクへの変更

## 配布方法

**プロジェクト内利用（このリポジトリclone済みの人）:**  
何もしない。Claude Codeを起動するだけで `/deadline-triage` が使える。

**外部メンバー用（cloneなし）:**
1. このSKILL.mdをコピー
2. `~/.claude/skills/deadline-triage/SKILL.md` に保存
3. `claude mcp add task-core --transport http https://tasuke.app/api/mcp --header "Authorization: Bearer tsk_xxx"` でMCP接続
4. Claude Code再起動 → `/deadline-triage` で起動

## 使い方の例

```
ユーザー: /deadline-triage

あなた: ワークスペースの状況を取得します...
       [並列でMCPツール呼び出し]

       ## 全体ヘルス
       - 期限切れ: 15件
       - 今週P0/P1集中: 戸野塚さんに6件
       - 進行中: 9件

       ## 🚨 緊急対応推奨（今日〜3日）
       | タスク | 担当 | 期限 → 提案 | 優先度 → 提案 | 理由 |
       |---|---|---|---|---|
       | TubeAI修正 | 戸野塚 | 3/18→今日 | P1→P0 | 既に1.5ヶ月超過 |
       ...

       全部承認しますか？個別確認しますか？

ユーザー: 個別

あなた: 1/15: 「TubeAI修正」P1→P0、期限を今日に。よろしいですか？
ユーザー: OK
あなた: 適用しました。2/15: ...
```
