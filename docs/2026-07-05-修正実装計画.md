## Phase 1: セキュリティ緊急 (即日対応推奨)

> `canAccessProject` (`src/lib/project-access.ts:8`) と `getAccessibleProjectIds` (同`:36-64`) は**既に存在する**。各ルートに1行加えるだけの外科的修正で塞げる。

### P1-1 REST タスク系ルートの認可漏れ (IDOR) `SEC-H1` / `FUNC-H1`

**目的**: 下記ルートは一様に
```ts
where: { id: params.id,
  project: { workspace: { members: { some: { userId: user.id } } } } }
```
で認可しており、「タスクが属する**ワークスペースのメンバーか**」しか見ていない。`Project.isPrivate` と `ProjectMember` を検証していないため、**同一WSの一般 MEMBER/VIEWER が非公開プロジェクトのタスクを閲覧・改変・削除できる**。対照的に `projects/[id]/tasks/route.ts:12` は `canAccessProject()` を使っており、REST個別ルートだけが防御を欠く。

**攻撃シナリオ**: `GET /api/tasks/search?q=評価&workspaceSlug=...` が非公開プロジェクトのタスク(タイトル・説明・**タスクID**)を返す → 得たIDで `GET/PATCH/DELETE /api/tasks/[id]`。ID推測不要。

**対象ルート (canAccessProject / getAccessibleProjectIds を欠く)**:
- `src/app/api/tasks/[id]/route.ts:14-17` (GET / PATCH / DELETE)
- `src/app/api/tasks/search/route.ts:36-42` (**ID漏洩の起点**)
- `src/app/api/tasks/[id]/comments/route.ts:11-14, 40-49`
- `src/app/api/tasks/[id]/subtasks/route.ts:15-19, 38-44`
- `src/app/api/tasks/[id]/labels/route.ts`
- `src/app/api/tasks/[id]/attachments/route.ts:12-20`
- `src/app/api/tasks/[id]/activities/route.ts:10-15`
- `src/app/api/tasks/[id]/assignees/route.ts:16-22`
- `src/app/api/tasks/[id]/move/route.ts`
- `src/app/api/tasks/bulk/route.ts:24-30`

**修正**:
- 個別タスクルート: タスク取得後に `projectId` を得て
  ```ts
  if (!(await canAccessProject(user.id, task.projectId))) {
    return errorResponse('アクセス権がありません', 403)
  }
  ```
  を所属チェック直後に追加。
- `search`: 結果を `getAccessibleProjectIds(user.id, workspaceId)` の集合で絞り込む (MCP版 `handleTaskSearch` と同じロジック)。
- `bulk`: 対象タスク群の `projectId` を集めて全て `canAccessProject` を通過するもののみ処理。1件でも権限外があれば403か、権限外を除外 (仕様は「除外」を推奨、UIの一括選択と整合)。

**verify**:
- ユニット/統合テスト: 「WSメンバーだが非公開プロジェクトの非メンバー」ユーザーで上記各エンドポイントを叩き、GET=403(または検索結果0件)、PATCH/DELETE=403 を確認。
- 正常系: プロジェクトメンバーなら200を維持。
- **この修正は SEC-H1 と FUNC-H1 を同時に解決する** (同一根本原因)。

### P2-5 手動/LINE議事録の重複抽出ガード `FUNC-M1`
